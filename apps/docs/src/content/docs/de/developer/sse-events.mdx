Scrape Dojo nutzt Server-Sent Events (SSE) für Echtzeit-Updates während Scrape-Ausführungen. Kein Polling, kein WebSocket-Overhead — einfach ein offener HTTP-Stream, über den das Backend dir sagt, was gerade passiert. Hier erfährst du alles über die Event-Typen, das Payload-Format und wie du SSE in deine eigenen Anwendungen integrierst.

## Endpoint

```
GET /api/events
```

Der SSE-Endpoint liefert einen Standard-EventStream (`text/event-stream`). Die Verbindung bleibt offen und der Server sendet Events, sobald sie auftreten.

## Verbindung herstellen

### Ohne Authentifizierung (Development)

```javascript
const eventSource = new EventSource('/api/events');

eventSource.onmessage = (event) => {
  const data = JSON.parse(event.data);
  console.log(`[${data.type}] ${data.scrapeId}: ${data.message}`);
};

eventSource.onerror = (error) => {
  console.error('SSE Verbindungsfehler:', error);
};
```

### Mit Authentifizierung

Da die EventSource-API keine Custom Headers unterstützt, wird das JWT-Token als Query-Parameter übergeben:

```javascript
const eventSource = new EventSource('/api/events?access_token=YOUR_JWT_TOKEN');

eventSource.onmessage = (event) => {
  const data = JSON.parse(event.data);
  console.log(`[${data.type}] ${data.scrapeId}: ${data.message}`);
};
```

<Aside type="note">
Access Tokens haben eine kurze Lebensdauer (Standard: 15 Minuten). Plane automatische Reconnects ein, wenn der Token abläuft!
</Aside>

## Event-Typen

| Typ | Beschreibung |
|-----|-------------|
| `scrape-start` | Scrape-Ausführung gestartet |
| `scrape-end` | Scrape abgeschlossen oder fehlgeschlagen |
| `step-status` | Step-Status geändert (running/completed/error) |
| `action-status` | Action-Status geändert |
| `loop-iteration` | Loop-Fortschritt (Index, Total, aktueller Wert) |
| `otp-required` | OTP-Eingabe vom Benutzer benötigt |
| `otp-received` | OTP-Code empfangen und verarbeitet |
| `log` | Anwendungs-Log (verschiedene Level) |
| `notification` | Benutzer-Benachrichtigung |
| `error` | Allgemeiner Fehler |
| `config-reload` | Konfiguration geändert (nur im Development-Modus) |

## ScrapeEvent Interface

Alle Events folgen diesem TypeScript-Interface:

```typescript
interface ScrapeEvent {
  type: string;
  scrapeId: string;
  runId?: string;
  stepName?: string;
  stepIndex?: number;
  actionName?: string;
  actionIndex?: number;
  actionType?: string;
  status?: 'pending' | 'running' | 'completed' | 'error' | 'waiting' | 'skipped';
  message?: string;
  error?: string;
  result?: unknown;
  loopName?: string;
  loopIndex?: number;
  loopTotal?: number;
  loopValue?: string;
  loopPath?: Array<{ name: string; index: number }>;
  timestamp: number;
  logLevel?: 'log' | 'error' | 'warn' | 'debug' | 'verbose';
  notification?: NotificationData;
  variables?: {
    runtime?: Record<string, any>;
    database?: Record<string, any>;
    final?: Record<string, any>;
  };
}
```

### Feld-Beschreibungen

| Feld | Beschreibung |
|------|-------------|
| `type` | Event-Typ (siehe Tabelle oben) |
| `scrapeId` | ID des betroffenen Scrapes |
| `runId` | ID des aktuellen Runs |
| `stepName` / `stepIndex` | Name und Index des aktuellen Steps |
| `actionName` / `actionIndex` / `actionType` | Details zur aktuellen Action |
| `status` | Aktueller Status des Elements |
| `message` | Menschenlesbare Nachricht |
| `error` | Fehlermeldung (bei Fehlern) |
| `result` | Ergebnis einer Action (bei Abschluss) |
| `loopName` / `loopIndex` / `loopTotal` / `loopValue` | Loop-Informationen |
| `loopPath` | Verschachtelte Loop-Hierarchie mit Name und Index |
| `timestamp` | Unix-Zeitstempel in Millisekunden |
| `logLevel` | Schweregrad bei Log-Events |
| `notification` | Benachrichtigungsdaten (Titel, Nachricht, Typ) |
| `variables` | Aktuelle Variablen (Runtime, Datenbank, zusammengeführt) |

## Integrationsbeispiele

### Vanilla JS — Event-Listener

```javascript
const eventSource = new EventSource('/api/events');

eventSource.onmessage = (event) => {
  const data = JSON.parse(event.data);

  switch (data.type) {
    case 'scrape-start':
      console.log(`Scrape ${data.scrapeId} gestartet (Run: ${data.runId})`);
      break;

    case 'step-status':
      console.log(`Step "${data.stepName}": ${data.status}`);
      break;

    case 'action-status':
      console.log(`Action "${data.actionName}" [${data.actionType}]: ${data.status}`);
      break;

    case 'loop-iteration':
      console.log(`Loop "${data.loopName}": ${data.loopIndex + 1}/${data.loopTotal}`);
      break;

    case 'log':
      console.log(`[${data.logLevel}] ${data.message}`);
      break;

    case 'scrape-end':
      console.log(`Scrape ${data.scrapeId} beendet: ${data.status}`);
      if (data.error) console.error(`Fehler: ${data.error}`);
      break;

    case 'otp-required':
      console.log('OTP-Eingabe erforderlich!');
      break;

    case 'notification':
      console.log(`Benachrichtigung: ${data.notification?.title}`);
      break;
  }
};
```

### Angular Service

```typescript
@Injectable({ providedIn: 'root' })
export class ScrapeEventsService {
  private eventSource: EventSource | null = null;
  private eventsSubject = new Subject<ScrapeEvent>();

  readonly events$ = this.eventsSubject.asObservable();

  connect(token: string): void {
    this.eventSource = new EventSource(
      `/api/events?access_token=${token}`
    );

    this.eventSource.onmessage = (event) => {
      const data: ScrapeEvent = JSON.parse(event.data);
      this.eventsSubject.next(data);
    };

    this.eventSource.onerror = () => {
      this.reconnect(token);
    };
  }

  disconnect(): void {
    this.eventSource?.close();
    this.eventSource = null;
  }

  private reconnect(token: string): void {
    this.disconnect();
    setTimeout(() => this.connect(token), 3000);
  }
}
```

### RxJS — Gefiltertes Event-Handling

```typescript
// Nur Events für einen bestimmten Scrape
this.eventsService.events$
  .pipe(
    filter(event => event.scrapeId === 'mein-scrape'),
    filter(event => ['scrape-start', 'scrape-end', 'step-status'].includes(event.type))
  )
  .subscribe(event => {
    this.updateUI(event);
  });
```

### Reconnection mit Exponential Backoff

```javascript
class SSEClient {
  constructor(url) {
    this.url = url;
    this.reconnectAttempts = 0;
    this.maxReconnectAttempts = 5;
  }

  connect() {
    this.eventSource = new EventSource(this.url);

    this.eventSource.onopen = () => {
      this.reconnectAttempts = 0; // Reset bei erfolgreicher Verbindung
    };

    this.eventSource.onmessage = (event) => {
      const data = JSON.parse(event.data);
      this.handleEvent(data);
    };

    this.eventSource.onerror = () => {
      this.eventSource.close();

      if (this.reconnectAttempts < this.maxReconnectAttempts) {
        const delay = Math.min(1000 * Math.pow(2, this.reconnectAttempts), 30000);
        console.log(`Reconnect in ${delay}ms (Versuch ${this.reconnectAttempts + 1})`);
        setTimeout(() => {
          this.reconnectAttempts++;
          this.connect();
        }, delay);
      } else {
        console.error('Max Reconnect-Versuche erreicht');
      }
    };
  }

  handleEvent(data) {
    // Event-Verarbeitung
  }
}
```

## Event-Flow Sequenz

Ein typischer Scrape-Durchlauf erzeugt Events in dieser Reihenfolge:

```
1. scrape-start          → Scrape wird gestartet
2. step-status (running) → Erster Step beginnt
3. action-status (running)    → Erste Action beginnt
4. action-status (completed)  → Erste Action abgeschlossen
5. ... weitere action-status Events
6. loop-iteration        → Bei Loops: für jede Iteration
7. step-status (completed)    → Step abgeschlossen
8. ... weitere Steps
9. scrape-end            → Scrape beendet (completed/error)
```

Bei Fehlern wird der `status` auf `error` gesetzt und das `error`-Feld enthält die Fehlermeldung.

## Testing via curl

```bash
# Terminal 1: SSE-Verbindung öffnen
curl -N -H "Accept: text/event-stream" \
  "http://localhost:3000/api/events?access_token=$TOKEN"

# Terminal 2: Scrape starten
curl -X POST http://localhost:3000/api/scrape/my-workflow \
  -H "Authorization: Bearer $TOKEN"
```

### Erwartete Ausgabe

```
data: {"type":"scrape-start","scrapeId":"my-workflow","runId":"run-abc123","timestamp":1704067200000}

data: {"type":"step-status","scrapeId":"my-workflow","stepName":"Extract","status":"running","timestamp":1704067201000}

data: {"type":"action-status","scrapeId":"my-workflow","actionName":"open","actionType":"navigate","status":"completed","timestamp":1704067202000}

data: {"type":"log","scrapeId":"my-workflow","message":"Navigiert zu https://example.com","logLevel":"log","timestamp":1704067203000}

data: {"type":"scrape-end","scrapeId":"my-workflow","status":"completed","timestamp":1704067210000}
```

<Aside type="tip">
SSE-Verbindungen sind unidirektional (Server → Client). Für Load-Balancing-Szenarien sind Sticky Sessions erforderlich, damit die Verbindung immer zum gleichen Server geht.
</Aside>
