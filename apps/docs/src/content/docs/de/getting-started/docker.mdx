Scrape Dojo wird als einzelnes Docker-Image ausgeliefert, das sowohl API als auch UI bündelt. Dieser Guide behandelt die Deployment-Konfiguration mit Fokus auf **Dateiberechtigungs-Management** — essenziell beim Betrieb auf NAS-Systemen wie Unraid, Synology oder TrueNAS.

## PUID / PGID — User- und Group-IDs

Docker-Container führen Prozesse mit internen User-IDs aus. Wenn der Container Dateien auf gemountete Volumes schreibt (Datenbank, Downloads, Logs), gehören diese Dateien dem internen Container-User. Wenn die IDs nicht mit deinem Host-System übereinstimmen, bekommst du Berechtigungsfehler.

**PUID** (Process User ID) und **PGID** (Process Group ID) ermöglichen es dir, den Container-User an deinen Host-User anzugleichen, damit die Datei-Ownership auf beiden Seiten korrekt ist.

### So funktioniert es

Beim Container-Start führt das Entrypoint-Script folgende Schritte aus:

<Steps>
1. Liest die Umgebungsvariablen `PUID` und `PGID`
2. Passt den internen `pptruser`-Account an diese IDs an
3. Korrigiert die Ownership aller Anwendungsverzeichnisse
4. Startet die Anwendung mit der aktualisierten Identität
</Steps>

### Eigene IDs herausfinden

Führe diesen Befehl auf deinem **Host-System** aus, um deine User- und Group-IDs zu finden:

```bash
id $(whoami)
# Beispielausgabe: uid=1000(user) gid=1000(user) groups=1000(user)
```

### Gängige NAS-Defaults

| System       | Standard PUID | Standard PGID | Hinweise                                    |
| ------------ | ------------- | ------------- | ------------------------------------------- |
| **Unraid**   | `99`          | `100`         | `nobody:users` — Unraids Standard-User      |
| **Synology** | `1026`        | `100`         | Erster auf dem NAS erstellter Benutzer       |
| **TrueNAS**  | `568`         | `568`         | `apps:apps` — TrueNAS SCALE Apps-User        |
| **Linux**    | `1000`        | `1000`        | Erster regulärer User (Ubuntu, Debian, etc.) |

## UMASK — Dateiberechtigungs-Maske

Die `UMASK`-Variable steuert die Standard-Berechtigungen für neu erstellte Dateien und Verzeichnisse im Container. Sie bestimmt, wer Dateien lesen, schreiben oder ausführen kann, die Scrape Dojo zur Laufzeit erstellt (Datenbank-Dateien, heruntergeladene Inhalte, Logs usw.).

| UMASK | Dateiberechtigungen | Verzeichnisberechtigungen | Anwendungsfall                         |
| ----- | ------------------- | ------------------------- | -------------------------------------- |
| `022` | `644` (rw-r--r--)   | `755` (rwxr-xr-x)        | **Standard** — Besitzer Vollzugriff, andere nur lesen |
| `002` | `664` (rw-rw-r--)   | `775` (rwxrwxr-x)        | Gemeinsamer Zugriff — Besitzer und Gruppe können schreiben |
| `077` | `600` (rw-------)   | `700` (rwx------)        | Restriktiv — nur Besitzer hat Zugriff  |

<Aside type="tip">
  Verwende `UMASK=002`, wenn andere Prozesse oder Benutzer in der gleichen Gruppe die Datendateien von Scrape Dojo lesen und schreiben müssen.
  Das ist auf NAS-Systemen üblich, wo mehrere Dienste Datenverzeichnisse teilen.
</Aside>

## Docker Compose Beispiele

### Standard-Setup

```yaml
services:
  scrape-dojo:
    image: ghcr.io/disane87/scrape-dojo:latest
    container_name: scrape-dojo
    restart: unless-stopped
    ports:
      - '8080:80'
    environment:
      - PUID=1000
      - PGID=1000
      - UMASK=022
      - SCRAPE_DOJO_ENCRYPTION_KEY=${SCRAPE_DOJO_ENCRYPTION_KEY}
    volumes:
      - ./data:/home/pptruser/app/data
      - ./downloads:/home/pptruser/app/downloads
      - ./logs:/home/pptruser/app/logs
      - ./config:/home/pptruser/app/config
```

### Unraid

```yaml
services:
  scrape-dojo:
    image: ghcr.io/disane87/scrape-dojo:latest
    container_name: scrape-dojo
    restart: unless-stopped
    ports:
      - '8080:80'
    environment:
      - PUID=99
      - PGID=100
      - UMASK=002
      - SCRAPE_DOJO_ENCRYPTION_KEY=${SCRAPE_DOJO_ENCRYPTION_KEY}
    volumes:
      - /mnt/user/appdata/scrape-dojo/data:/home/pptruser/app/data
      - /mnt/user/appdata/scrape-dojo/downloads:/home/pptruser/app/downloads
      - /mnt/user/appdata/scrape-dojo/logs:/home/pptruser/app/logs
      - /mnt/user/appdata/scrape-dojo/config:/home/pptruser/app/config
```

<Aside type="note">
  Auf Unraid entsprechen `PUID=99` und `PGID=100` dem `nobody:users`-Account — dem Standard-User für Docker-Container.
  `UMASK=002` stellt sicher, dass Dateien für die Gruppe beschreibbar sind, was beim Teilen von Daten zwischen Containern wichtig ist.
</Aside>

### Synology

```yaml
services:
  scrape-dojo:
    image: ghcr.io/disane87/scrape-dojo:latest
    container_name: scrape-dojo
    restart: unless-stopped
    ports:
      - '8080:80'
    environment:
      - PUID=1026
      - PGID=100
      - UMASK=002
      - SCRAPE_DOJO_ENCRYPTION_KEY=${SCRAPE_DOJO_ENCRYPTION_KEY}
    volumes:
      - /volume1/docker/scrape-dojo/data:/home/pptruser/app/data
      - /volume1/docker/scrape-dojo/downloads:/home/pptruser/app/downloads
      - /volume1/docker/scrape-dojo/logs:/home/pptruser/app/logs
      - /volume1/docker/scrape-dojo/config:/home/pptruser/app/config
```

## Volumes

Scrape Dojo verwendet folgende persistente Verzeichnisse:

| Container-Pfad                      | Zweck                                       | Kritisch? |
| ----------------------------------- | ------------------------------------------- | --------- |
| `/home/pptruser/app/data`           | Datenbank (SQLite) und Anwendungsdaten      | **Ja** |
| `/home/pptruser/app/downloads`      | Heruntergeladene Dateien aus Scrape-Actions | Ja |
| `/home/pptruser/app/logs`           | Anwendungs- und Supervisor-Logs            | Nein |
| `/home/pptruser/app/browser-data`   | Chromium Browser-Profildaten               | Empfohlen |
| `/home/pptruser/app/config`         | Scrape-Konfigurationsdateien (`.jsonc`)    | **Ja** |

<Aside type="danger" title="Datenpersistenz">
  **Ohne Volume-Mounts gehen alle Daten verloren**, wenn der Container entfernt oder neu erstellt wird (z.B. bei einem Image-Update).
  Mounte mindestens `data` und `config`, um deine Datenbank und Scrape-Konfigurationen zu sichern.
</Aside>

### Bind Mounts vs. Named Volumes

Du kannst entweder **Bind Mounts** oder **Named Docker Volumes** zur Datenpersistierung verwenden:

| Methode | Syntax | Vorteile | Nachteile |
| --- | --- | --- | --- |
| **Bind Mount** | `./data:/home/pptruser/app/data` | Dateien auf dem Host sichtbar, einfach zu sichern | Berechtigungen muessen verwaltet werden (PUID/PGID) |
| **Named Volume** | `scrape-data:/home/pptruser/app/data` | Docker verwaltet den Speicher, keine Berechtigungsprobleme | Nicht direkt auf dem Host sichtbar |

<Aside type="tip">
  **Bind Mounts** sind fuer die meisten Setups empfohlen — besonders auf NAS-Systemen — weil du die Dateien direkt sehen und sichern kannst.
  Verwende Named Volumes nur, wenn du keinen direkten Dateizugriff brauchst.
</Aside>

## Datenbank-Synchronisation

`DB_SYNCHRONIZE` steuert, ob TypeORM beim Start automatisch Datenbanktabellen erstellt und aktualisiert.

| Szenario | Wert | Hinweise |
| --- | --- | --- |
| **Erster Start** | `true` | **Erforderlich** — erstellt alle Datenbanktabellen |
| **Nach der Ersteinrichtung** | `false` | Verhindert versehentliche Schema-Aenderungen oder Datenverlust bei Updates |

<Aside type="caution" title="Wichtig: DB_SYNCHRONIZE=false nach dem ersten Start setzen">
  Nachdem die Ersteinrichtung abgeschlossen ist und die Datenbank erstellt wurde, setze `DB_SYNCHRONIZE=false` in deiner Umgebung.
  Wenn es in Produktion auf `true` bleibt, kann es bei Updates zu Datenverlust kommen,
  da TypeORM Spalten loeschen und neu erstellen kann, wenn sich Entity-Definitionen aendern.
</Aside>

## Umgebungsvariablen

Im `environment`-Block deiner `docker-compose.yml` kannst du das Verhalten von Scrape Dojo steuern. Hier die wichtigsten:

| Variable | Standard | Beschreibung |
| --- | --- | --- |
| `SCRAPE_DOJO_ENCRYPTION_KEY` | — | **Pflicht.** 256-Bit Hex-Key fuer die Verschluesselung von Secrets |
| `DB_TYPE` | `sqlite` | Datenbank-Typ: `sqlite`, `mysql`, `postgres` |
| `DB_SYNCHRONIZE` | `true` | Erstellt Tabellen automatisch beim Start. Nach der Ersteinrichtung auf `false` setzen (siehe oben) |
| `SCRAPE_DOJO_AUTH_ENABLED` | `true` | Authentifizierung global aktivieren/deaktivieren |
| `SCRAPE_DOJO_AUTH_JWT_SECRET` | — | JWT-Signing-Secret (mind. 32 Zeichen, in Produktion mit Auth erforderlich) |
| `SCRAPE_DOJO_AUTH_REFRESH_TOKEN_SECRET` | — | Refresh-Token-Secret (mind. 32 Zeichen) |
| `SCRAPE_DOJO_AUTH_REQUIRE_MFA` | `true` | MFA/TOTP fuer alle Nutzer erzwingen |
| `SCRAPE_DOJO_PORT` | `3000` | Interner API-Port (in Docker normalerweise nicht aendern) |
| `SCRAPE_DOJO_CORS_ORIGIN` | — | CORS-Origins (kommasepariert, fuer Produktion empfohlen) |
| `SCRAPE_DOJO_AUTH_OIDC_ENABLED` | `false` | OIDC/SSO-Login aktivieren (Keycloak, Auth0, Google, Azure AD) |
| `PUID` | `1000` | User-ID fuer den Container-Prozess — passend zum Host-User setzen |
| `PGID` | `1000` | Group-ID fuer den Container-Prozess |
| `UMASK` | `022` | Dateiberechtigungs-Maske fuer neu erstellte Dateien |

<Aside type="tip">
  Die vollstaendige Referenz mit allen Optionen (Datenbank, Auth, MFA, OIDC, Logging, Rate Limiting) findest du unter
  [Umgebungsvariablen](/de/developer/environment-variables/).
</Aside>

**Beispiel: Produktions-Setup mit Auth und MySQL:**

```yaml
services:
  scrape-dojo:
    image: ghcr.io/disane87/scrape-dojo:latest
    ports:
      - '8080:80'
    environment:
      - SCRAPE_DOJO_ENCRYPTION_KEY=dein_64_zeichen_hex_key
      - DB_TYPE=mysql
      - DB_HOST=db
      - DB_PORT=3306
      - DB_USERNAME=scrape_dojo
      - DB_PASSWORD=sicheres_passwort
      - DB_DATABASE=scrape_dojo
      - DB_SYNCHRONIZE=false
      - SCRAPE_DOJO_AUTH_ENABLED=true
      - SCRAPE_DOJO_AUTH_JWT_SECRET=dein_jwt_secret_mind_32_zeichen
      - SCRAPE_DOJO_AUTH_REFRESH_TOKEN_SECRET=dein_refresh_secret
    volumes:
      - ./downloads:/home/pptruser/app/downloads
      - ./logs:/home/pptruser/app/logs
      - ./config:/home/pptruser/app/config
      - ./browser-data:/home/pptruser/app/browser-data
    restart: unless-stopped
```

## Fehlerbehebung

### Permission denied Fehler

Wenn du `EACCES` oder `Permission denied` Fehler in den Logs siehst:

1. Prüfe, ob `PUID` und `PGID` mit deinem Host-User übereinstimmen: `id $(whoami)`
2. Überprüfe die Verzeichnis-Ownership auf dem Host: `ls -la ./data`
3. Falls nötig, korrigiere die Ownership manuell: `sudo chown -R <PUID>:<PGID> ./data ./downloads ./logs ./config`

### Dateien mit falschen Berechtigungen erstellt

Wenn andere Dienste oder Benutzer nicht auf von Scrape Dojo erstellte Dateien zugreifen können:

1. Setze `UMASK=002` um Gruppen-Schreibzugriff zu ermöglichen
2. Stelle sicher, dass der andere Dienst die gleiche `PGID` verwendet
