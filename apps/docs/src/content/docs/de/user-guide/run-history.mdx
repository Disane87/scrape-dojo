Jeder Scrape, der in Scrape Dojo läuft, wird als "Run" protokolliert. Du bekommst eine lückenlose Historie mit allen Details -- was lief, wie lange es gedauert hat, was schiefgegangen ist und welche Daten extrahiert wurden. Dazu gibt's Echtzeit-Monitoring via SSE, damit du live zuschauen kannst.

Kurz gesagt: Kein Scrape-Run geht verloren, und wenn mal was klemmt, findest du hier die Antworten.

## 📋 Run-Felder

Jeder Run zeichnet folgende Informationen auf:

| Feld | Beschreibung |
|------|--------------|
| `id` | Eindeutige Run-ID |
| `scrapeId` | ID des ausgeführten Scrapes |
| `status` | `running`, `completed`, `error` oder `aborted` |
| `trigger` | Wie der Run gestartet wurde: `manual`, `scheduled` oder `api` |
| `startTime` | Startzeitpunkt |
| `endTime` | Endzeitpunkt |
| `error` | Fehlermeldung (bei Status `error`) |
| `steps` | Liste aller ausgeführten Steps mit deren Actions |
| `logs` | Protokollierte Log-Einträge |

### 🚦 Run-Status

| Status | Bedeutung |
|--------|-----------|
| `running` | Wird aktuell ausgeführt |
| `completed` | Erfolgreich abgeschlossen |
| `error` | Mit Fehler beendet |
| `aborted` | Manuell abgebrochen |

## 🖥️ Run History in der UI

Die UI zeigt dir eine Liste aller Runs mit Status, Dauer und Trigger-Typ. Ein Klick auf einen Run öffnet die Detailansicht.

### 🔍 Detailansicht

In der Detailansicht siehst du:
- Vollständige Step-Hierarchie (einschließlich verschachtelter Loops)
- Parameter und Ergebnisse jeder Action
- Fehlermeldungen mit Kontext
- Artefakte (extrahierte Daten, Screenshots)
- Runtime-Variablen und deren Werte

## 🌐 Run-Management via API

### 📥 Alle Runs abrufen

```bash
# Alle Runs (mit Limit)
GET /api/runs?limit=50

# Runs für einen bestimmten Scrape
GET /api/runs?scrapeId=my-workflow&limit=100
```

**Response:**
```json
[
  {
    "id": "run-1704067200000-abc123",
    "scrapeId": "my-workflow",
    "status": "completed",
    "trigger": "manual",
    "startTime": "2025-01-11T10:00:00.000Z",
    "endTime": "2025-01-11T10:05:30.000Z",
    "duration": 330000,
    "error": null,
    "stepCount": 5,
    "actionCount": 23
  }
]
```

### 📥 Einzelnen Run abrufen

```bash
GET /api/runs/{runId}
```

Gibt den vollständigen Run mit allen Steps, Actions, Parametern und Ergebnissen zurück.

## 🐛 Debug-Daten

Der Debug-Endpoint ist dein bester Freund bei der Fehlersuche. Er liefert dir alle internen Daten eines Runs:

```bash
GET /api/runs/{runId}/debug
```

**Response:**
```json
{
  "run": {
    "id": "run-abc123",
    "status": "completed",
    "steps": ["..."]
  },
  "previousData": {
    "open-page": null,
    "extract-title": {
      "title": "Beispielseite",
      "subtitle": "Willkommen"
    },
    "transform-data": {
      "formatted": "Beispielseite - Willkommen"
    }
  },
  "storedData": {
    "lastUpdate": "2025-01-10",
    "cache": {}
  },
  "variables": {
    "targetYear": 2025,
    "region": "de"
  }
}
```

Die `previousData`-Sektion zeigt das Ergebnis jeder Action anhand ihres Namens. So kannst du nachvollziehen, welche Daten an jedem Punkt des Workflows verfügbar waren.

## 📦 Artefakte

Artefakte sind extrahierte Daten, die von `artifacts`-Actions während der Ausführung erzeugt wurden:

```bash
GET /api/runs/{runId}/artifacts
```

Artefakte werden in der UI als Karten, Tabellen oder JSON-Ansichten dargestellt, je nach Datentyp.

## 🧹 Cleanup

Damit deine Datenbank nicht überquillt, gibt es verschiedene Cleanup-Optionen.

### 🗑️ Alte Runs löschen

```bash
POST /api/runs/cleanup
Content-Type: application/json

{
  "days": 30
}
```

Löscht alle Runs, die älter als die angegebene Anzahl an Tagen sind.

**Response:**
```json
{
  "deleted": 142,
  "message": "Deleted 142 runs older than 30 days"
}
```

### 🗑️ Einzelnen Run löschen

```bash
DELETE /api/runs/{runId}
```

### 🗑️ Alle Runs eines Scrapes löschen

```bash
DELETE /api/scrapes/{scrapeId}/runs
```

### 🔍 Selektives Cleanup

```bash
# Nur fehlerhafte Runs löschen (älter als 7 Tage)
POST /api/runs/cleanup
{
  "days": 7,
  "status": "error"
}
```

<Aside type="tip">
Richte einen Cleanup-Zeitplan ein, z.B. "Runs älter als 30 Tage löschen". Gerade in Entwicklungsumgebungen sammeln sich schnell hunderte Runs an.
</Aside>

## 📡 Echtzeit-Monitoring via SSE

Während ein Scrape läuft, liefern Server-Sent Events (SSE) Live-Status-Updates direkt in die UI:

| Event | Beschreibung |
|-------|--------------|
| `scrape-start` / `scrape-end` | Beginn und Ende des gesamten Scrapes |
| `step-status` | Statusänderungen einzelner Steps |
| `action-status` | Statusänderungen einzelner Actions |
| `loop-iteration` | Fortschritt bei Loop-Actions |
| `log` | Log-Nachrichten der Ausführung |

Die UI aktualisiert sich automatisch anhand dieser Events und zeigt den Fortschritt in Echtzeit an.

<Aside type="note">
Details zur SSE-Integration für eigene Clients findest du unter [SSE Events](/de/developer/sse-events/).
</Aside>

## 🔍 Fehleranalyse

Hier sind die typischen Szenarien, denen du begegnen wirst, und wie du sie angehst:

### ⚠️ Action gibt falsches Ergebnis

1. Debug-Daten abrufen: `GET /api/runs/{runId}/debug`
2. In `previousData` den Wert der betroffenen Action prüfen
3. Selector, Transform-Expression oder Parameter anpassen

### ⚠️ Variable nicht verfügbar

1. Debug-Daten prüfen -- ist die Variable im `variables`-Objekt vorhanden?
2. Variablen-Definition in `metadata.variables` prüfen
3. Auflösungsreihenfolge beachten (Runtime > Database > Config Default)

### ⚠️ Loop-Iteration fehlgeschlagen

1. In der Detailansicht den `loopPath` prüfen (zeigt verschachtelte Position)
2. `currentData` an der Fehlerstelle analysieren
3. Optional: `optional: true` für fehlertolerante Extraktion verwenden

### ⚠️ Timeout-Fehler

1. Prüfen, ob der Selector korrekt ist und das Element existiert
2. `waitForSelector`-Action vor der betroffenen Action hinzufügen
3. Timeout-Wert erhöhen, falls die Seite langsam lädt

## ✅ Best Practices

- **Logger-Actions einsetzen** für wichtige Checkpoints im Workflow:
  ```jsonc
  {
    "name": "log-progress",
    "action": "logger",
    "params": {
      "message": "Verarbeitet: {{previousData.items.length}} Einträge"
    }
  }
  ```

- **Artefakte speichern** bei komplexen Workflows für Zwischenergebnisse:
  ```jsonc
  {
    "name": "save-intermediate",
    "action": "artifacts",
    "params": {
      "title": "Zwischenergebnis",
      "data": "{{previousData.extractedData}}"
    }
  }
  ```

- **Retention Policy** konfigurieren -- kürzere Aufbewahrung in Entwicklungsumgebungen, längere in Produktion

- **Fehlertolerante Extraktion** mit `optional: true` verwenden, wenn Elemente nicht immer vorhanden sind
