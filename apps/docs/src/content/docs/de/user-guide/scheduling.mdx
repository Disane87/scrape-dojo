Klar, du kannst deine Scrapes jedes Mal manuell starten. Aber mal ehrlich -- das nervt auf Dauer. Scrape Dojo bietet dir verschiedene Trigger-Typen, mit denen deine Scrapes automatisch laufen. Ob zeitgesteuert per Cron, per Webhook von außen oder direkt beim App-Start.

Einmal einrichten und dann läuft die Sache von alleine. So soll das sein!

## 📋 Trigger-Typen

| Typ | Beschreibung |
|-----|--------------|
| `manual` | Nur manuelle Ausführung über UI oder API |
| `cron` | Zeitgesteuerte Ausführung nach Cron-Expression |
| `webhook` | Auslösung durch externen HTTP-Request |
| `startup` | Automatische Ausführung beim Anwendungsstart |

Trigger werden im `metadata.triggers`-Array der Scrape-Konfiguration definiert:

```jsonc
{
  "scrapes": [
    {
      "id": "daily-report",
      "metadata": {
        "description": "Täglicher Bericht",
        "version": "1.0.0",
        "triggers": [
          { "type": "manual" },
          { "type": "cron" }
        ]
      },
      "steps": [
        // ...
      ]
    }
  ]
}
```

## ⏰ Cron-Konfiguration

Ein Scrape mit `"type": "cron"` im Triggers-Array kann über die UI oder API mit einer Cron-Expression versehen werden.

### 🔧 Schedule über die API konfigurieren

```bash
PUT /api/scrapes/{scrapeId}/schedule
Content-Type: application/json

{
  "scheduleEnabled": true,
  "cronExpression": "0 9 * * *",
  "timezone": "Europe/Berlin"
}
```

### 📥 Schedule abrufen

```bash
GET /api/scrapes/{scrapeId}/schedule
```

**Response:**
```json
{
  "scrapeId": "daily-report",
  "manualEnabled": true,
  "scheduleEnabled": true,
  "cronExpression": "0 9 * * *",
  "timezone": "Europe/Berlin",
  "lastScheduledRun": 1705824000000,
  "nextScheduledRun": 1705910400000
}
```

### 📋 Alle Schedules abrufen

```bash
GET /api/schedules
```

### 📡 Scheduler-Status prüfen

```bash
GET /api/scheduler/status
```

**Response:**
```json
[
  {
    "scrapeId": "daily-report",
    "cronExpression": "0 9 * * *",
    "nextRun": "2025-01-22T09:00:00.000Z"
  },
  {
    "scrapeId": "weekly-newsletter",
    "cronExpression": "0 8 * * 1",
    "nextRun": "2025-01-27T08:00:00.000Z"
  }
]
```

## 📝 Cron-Syntax

Scrape Dojo verwendet die Standard-POSIX-Cron-Syntax mit 5 Feldern:

```
Minute  Stunde  Tag(Monat)  Monat  Wochentag
  *       *        *          *       *
```

### 📋 Felder

| Feld | Wertebereich | Beschreibung |
|------|-------------|--------------|
| Minute | 0-59 | Minute der Ausführung |
| Stunde | 0-23 | Stunde der Ausführung |
| Tag (Monat) | 1-31 | Tag im Monat |
| Monat | 1-12 | Monat im Jahr |
| Wochentag | 0-7 | Wochentag (0 und 7 = Sonntag) |

### 🔣 Spezielle Zeichen

| Zeichen | Bedeutung | Beispiel |
|---------|-----------|----------|
| `*` | Jeder Wert | `* * * * *` = Jede Minute |
| `*/n` | Alle n Einheiten | `*/15 * * * *` = Alle 15 Minuten |
| `n-m` | Bereich | `0 8-18 * * *` = Stündlich von 8 bis 18 Uhr |
| `n,m` | Liste | `0,30 * * * *` = Zur vollen und halben Stunde |
| `n-m/x` | Bereich mit Schritt | `0 8-18/2 * * *` = Alle 2 Stunden von 8-18 Uhr |

### 💡 Beispiele

```bash
# Täglich um 9:00 Uhr
0 9 * * *

# Jeden Montag um 9:00 Uhr
0 9 * * 1

# Alle 30 Minuten
*/30 * * * *

# Stündlich zur vollen Stunde
0 * * * *

# Werktags um 6:00 Uhr (Montag bis Freitag)
0 6 * * 1-5

# Am 1. jeden Monats um Mitternacht
0 0 1 * *

# Alle 2 Stunden zwischen 8 und 18 Uhr
0 8-18/2 * * *
```

<Aside type="tip">
Du bist dir bei der Cron-Syntax nicht sicher? Schau auf [crontab.guru](https://crontab.guru/) vorbei -- da kannst du Ausdrücke live testen und bekommst eine menschenlesbare Beschreibung.
</Aside>

## 🌍 Zeitzone

Die Standard-Zeitzone ist `Europe/Berlin`. Du kannst sie pro Schedule individuell konfigurieren:

```bash
PUT /api/scrapes/{scrapeId}/schedule
{
  "scheduleEnabled": true,
  "cronExpression": "0 9 * * *",
  "timezone": "America/New_York"
}
```

Alle gängigen IANA-Zeitzonen werden unterstützt (z.B. `UTC`, `Europe/Berlin`, `America/New_York`, `Asia/Tokyo`).

<Aside type="caution">
Beachte bei der Zeitzone die Sommerzeit! `0 9 * * *` mit `timezone: "UTC"` ist in Deutschland 10:00 (MEZ) bzw. 11:00 (MESZ).
</Aside>

## 🪝 Webhook-Trigger

<Aside type="caution">
  Der Webhook-Trigger ist derzeit **geplant, aber noch nicht implementiert**. Der Trigger-Typ existiert im Konfigurationsschema, aber es gibt noch keinen API-Endpoint zum Empfangen von Webhook-Aufrufen. Dieser Abschnitt beschreibt das geplante Verhalten.
</Aside>

Der Webhook-Trigger wird das Starten von Scrapes durch externe HTTP-Requests ermöglichen:

```jsonc
{
  "metadata": {
    "triggers": [
      {
        "type": "webhook",
        "secret": "mein-geheimer-token",
        "methods": ["POST"]
      }
    ]
  }
}
```

Geplanter Aufruf:

```bash
curl -X POST http://localhost:3000/api/webhooks/{scrapeId} \
  -H "Content-Type: application/json" \
  -H "X-Webhook-Secret: mein-geheimer-token" \
  -d '{ "variables": { "orderId": "12345" } }'
```

<Aside type="note">
Über den Request-Body kannst du Variablen übergeben, die dann im Scrape als `{{variables.orderId}}` verfügbar sind. So machst du deine Webhooks dynamisch.
</Aside>

## 🚀 Startup-Trigger

Der Startup-Trigger führt einen Scrape automatisch beim Start der Anwendung aus:

```jsonc
{
  "metadata": {
    "triggers": [
      {
        "type": "startup",
        "config": {
          "delay": 5000
        }
      }
    ]
  }
}
```

Der optionale `delay`-Wert gibt die Verzögerung in Millisekunden nach dem Anwendungsstart an. Das ist nützlich, damit die App erst komplett hochfahren kann, bevor der Scrape losläuft.

## 🔗 Trigger kombinieren

Du kannst mehrere Trigger für denselben Scrape definieren -- kein Problem:

```jsonc
{
  "metadata": {
    "triggers": [
      { "type": "manual" },
      { "type": "cron" },
      {
        "type": "webhook",
        "config": {
          "webhookSecret": "mein-secret"
        }
      }
    ]
  }
}
```

Die Run-Historie zeigt dir an, welcher Trigger die Ausführung ausgelöst hat (`manual`, `scheduled`, `api`).

## ✅ Best Practices

- **Zeitzone explizit setzen**, besonders bei internationalen Teams
- **Intervalle nicht zu kurz wählen** -- mindestens 15 Minuten empfohlen
- **Überlappung vermeiden** -- stelle sicher, dass der Scrape vor der nächsten Ausführung beendet ist
- **Erst manuell testen**, dann Schedule aktivieren
- **Logs regelmäßig prüfen**, insbesondere nach Aktivierung eines neuen Schedules
- **Webhook-Secrets verwenden** und über HTTPS aufrufbar machen

## 🔍 Troubleshooting

### ⚠️ Schedule läuft nicht

1. Prüfen, ob `scheduleEnabled: true` gesetzt ist:
   ```bash
   GET /api/scrapes/{id}/schedule
   ```
2. Scheduler-Status prüfen:
   ```bash
   GET /api/scheduler/status
   ```
3. Cron-Expression validieren (z.B. auf [crontab.guru](https://crontab.guru/))
4. API-Logs auf Fehlermeldungen prüfen

### ⚠️ Falsche Ausführungszeit

1. Zeitzone der Schedule-Konfiguration prüfen
2. Serverzeit und -zeitzone verifizieren über `/api/health`
3. Beachten: `0 9 * * *` mit `timezone: "UTC"` ist in Deutschland 10:00 (MEZ) bzw. 11:00 (MESZ)
