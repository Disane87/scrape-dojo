Scrape Dojo ships as a single Docker image that bundles both the API and UI. This guide covers deployment configuration with a focus on **file permission management** — essential when running on NAS systems like Unraid, Synology, or TrueNAS.

## PUID / PGID — User and Group IDs

Docker containers run processes with internal user IDs. When the container writes files to mounted volumes (database, downloads, logs), these files are owned by the container's internal user. If the IDs don't match your host system, you'll run into permission errors.

**PUID** (Process User ID) and **PGID** (Process Group ID) let you align the container's user with your host user so that file ownership is correct on both sides.

### How it works

On container startup, the entrypoint script:

<Steps>
1. Reads the `PUID` and `PGID` environment variables
2. Modifies the internal `pptruser` account to match those IDs
3. Adjusts ownership of all application directories
4. Starts the application with the updated identity
</Steps>

### Finding your IDs

Run this command on your **host system** to find your user and group IDs:

```bash
id $(whoami)
# Example output: uid=1000(user) gid=1000(user) groups=1000(user)
```

### Common NAS defaults

| System       | Default PUID | Default PGID | Notes                                      |
| ------------ | ------------ | ------------ | ------------------------------------------ |
| **Unraid**   | `99`         | `100`        | `nobody:users` — Unraid's default user     |
| **Synology** | `1026`       | `100`        | First user created on the NAS              |
| **TrueNAS**  | `568`        | `568`        | `apps:apps` — TrueNAS SCALE apps user      |
| **Linux**    | `1000`       | `1000`       | First regular user (Ubuntu, Debian, etc.)  |

## UMASK — File Permission Mask

The `UMASK` variable controls the default permissions for newly created files and directories inside the container. It determines who can read, write, or execute files that Scrape Dojo creates at runtime (database files, downloaded content, logs, etc.).

| UMASK | File permissions | Directory permissions | Use case                         |
| ----- | ---------------- | --------------------- | -------------------------------- |
| `022` | `644` (rw-r--r--)| `755` (rwxr-xr-x)    | **Default** — owner full access, others read-only |
| `002` | `664` (rw-rw-r--)| `775` (rwxrwxr-x)    | Shared access — owner and group can write         |
| `077` | `600` (rw-------)| `700` (rwx------)    | Restrictive — only owner has access               |

<Aside type="tip">
  Use `UMASK=002` if you need other processes or users in the same group to read and write Scrape Dojo's data files.
  This is common on NAS systems where multiple services share data directories.
</Aside>

## Docker Compose examples

### Standard setup

```yaml
services:
  scrape-dojo:
    image: ghcr.io/disane87/scrape-dojo:latest
    container_name: scrape-dojo
    restart: unless-stopped
    ports:
      - '8080:80'
    environment:
      - PUID=1000
      - PGID=1000
      - UMASK=022
      - SCRAPE_DOJO_ENCRYPTION_KEY=${SCRAPE_DOJO_ENCRYPTION_KEY}
    volumes:
      - ./data:/home/pptruser/app/data
      - ./downloads:/home/pptruser/app/downloads
      - ./logs:/home/pptruser/app/logs
      - ./config:/home/pptruser/app/config
```

### Unraid

```yaml
services:
  scrape-dojo:
    image: ghcr.io/disane87/scrape-dojo:latest
    container_name: scrape-dojo
    restart: unless-stopped
    ports:
      - '8080:80'
    environment:
      - PUID=99
      - PGID=100
      - UMASK=002
      - SCRAPE_DOJO_ENCRYPTION_KEY=${SCRAPE_DOJO_ENCRYPTION_KEY}
    volumes:
      - /mnt/user/appdata/scrape-dojo/data:/home/pptruser/app/data
      - /mnt/user/appdata/scrape-dojo/downloads:/home/pptruser/app/downloads
      - /mnt/user/appdata/scrape-dojo/logs:/home/pptruser/app/logs
      - /mnt/user/appdata/scrape-dojo/config:/home/pptruser/app/config
```

<Aside type="note">
  On Unraid, `PUID=99` and `PGID=100` correspond to the `nobody:users` account — the default user for Docker containers.
  Setting `UMASK=002` ensures that files are group-writable, which is important when sharing data between containers.
</Aside>

## Volumes

Scrape Dojo uses the following persistent directories:

| Container path                      | Purpose                                    | Critical? |
| ----------------------------------- | ------------------------------------------ | --------- |
| `/home/pptruser/app/data`           | Database (SQLite) and application data     | **Yes** |
| `/home/pptruser/app/downloads`      | Downloaded files from scrape actions       | Yes |
| `/home/pptruser/app/logs`           | Application and supervisor logs            | No |
| `/home/pptruser/app/browser-data`   | Chromium browser profile data              | Recommended |
| `/home/pptruser/app/config`         | Scrape configuration files (`.jsonc`)      | **Yes** |

<Aside type="danger" title="Data persistence">
  **Without volume mounts, all data is lost** when the container is removed or recreated (e.g., during an image update).
  At minimum, always mount `data` and `config` to persist your database and scrape configurations.
</Aside>

### Bind mounts vs. named volumes

You can use either **bind mounts** or **named Docker volumes** to persist data:

| Method | Syntax | Pros | Cons |
| --- | --- | --- | --- |
| **Bind mount** | `./data:/home/pptruser/app/data` | Files visible on host, easy to backup | Must manage permissions (PUID/PGID) |
| **Named volume** | `scrape-data:/home/pptruser/app/data` | Docker manages storage, no permission issues | Not directly visible on host |

<Aside type="tip">
  **Bind mounts** are recommended for most setups — especially on NAS systems — because you can see and back up the files directly.
  Use named volumes only if you don't need direct file access.
</Aside>

## Database synchronization

`DB_SYNCHRONIZE` controls whether TypeORM automatically creates and updates database tables on startup.

| Scenario | Value | Notes |
| --- | --- | --- |
| **First start** | `true` | **Required** — creates all database tables |
| **After initial setup** | `false` | Prevents accidental schema changes or data loss on updates |

<Aside type="caution" title="Important: Set DB_SYNCHRONIZE=false after first start">
  After your initial setup is complete and the database has been created, set `DB_SYNCHRONIZE=false` in your environment.
  Leaving it on `true` in production can lead to data loss if entity definitions change during an update,
  because TypeORM may drop and recreate columns.
</Aside>

## Environment variables

Use the `environment` block in your `docker-compose.yml` to configure Scrape Dojo's behavior. Here are the most important ones:

| Variable | Default | Description |
| --- | --- | --- |
| `SCRAPE_DOJO_ENCRYPTION_KEY` | — | **Required.** 256-bit hex key for encrypting secrets |
| `DB_TYPE` | `sqlite` | Database type: `sqlite`, `mysql`, `postgres` |
| `DB_SYNCHRONIZE` | `true` | Auto-creates tables on startup. Set to `false` after initial setup (see above) |
| `SCRAPE_DOJO_AUTH_ENABLED` | `true` | Enable/disable authentication globally |
| `SCRAPE_DOJO_AUTH_JWT_SECRET` | — | JWT signing secret (min. 32 chars, required in production with auth enabled) |
| `SCRAPE_DOJO_AUTH_REFRESH_TOKEN_SECRET` | — | Refresh token secret (min. 32 chars) |
| `SCRAPE_DOJO_AUTH_REQUIRE_MFA` | `true` | Require MFA/TOTP for all users |
| `SCRAPE_DOJO_PORT` | `3000` | Internal API port (usually no need to change in Docker) |
| `SCRAPE_DOJO_CORS_ORIGIN` | — | CORS origins (comma-separated, recommended for production) |
| `SCRAPE_DOJO_AUTH_OIDC_ENABLED` | `false` | Enable OIDC/SSO login (Keycloak, Auth0, Google, Azure AD) |
| `PUID` | `1000` | User ID for the container process — set to match your host user |
| `PGID` | `1000` | Group ID for the container process |
| `UMASK` | `022` | File permission mask for newly created files |

<Aside type="tip">
  For the full reference with all options (database, auth, MFA, OIDC, logging, rate limiting), see
  [Environment Variables](/en/developer/environment-variables/).
</Aside>

**Example: Production setup with auth and MySQL:**

```yaml
services:
  scrape-dojo:
    image: ghcr.io/disane87/scrape-dojo:latest
    ports:
      - '8080:80'
    environment:
      - SCRAPE_DOJO_ENCRYPTION_KEY=your_64_char_hex_key
      - DB_TYPE=mysql
      - DB_HOST=db
      - DB_PORT=3306
      - DB_USERNAME=scrape_dojo
      - DB_PASSWORD=secure_password
      - DB_DATABASE=scrape_dojo
      - DB_SYNCHRONIZE=false
      - SCRAPE_DOJO_AUTH_ENABLED=true
      - SCRAPE_DOJO_AUTH_JWT_SECRET=your_jwt_secret_min_32_chars
      - SCRAPE_DOJO_AUTH_REFRESH_TOKEN_SECRET=your_refresh_secret
    volumes:
      - ./downloads:/home/pptruser/app/downloads
      - ./logs:/home/pptruser/app/logs
      - ./config:/home/pptruser/app/config
      - ./browser-data:/home/pptruser/app/browser-data
    restart: unless-stopped
```

## Troubleshooting

### Permission denied errors

If you see `EACCES` or `Permission denied` errors in the logs:

1. Check that `PUID` and `PGID` match your host user: `id $(whoami)`
2. Verify volume directory ownership on the host: `ls -la ./data`
3. If needed, fix ownership manually: `sudo chown -R <PUID>:<PGID> ./data ./downloads ./logs ./config`

### Files created with wrong permissions

If other services or users can't access files created by Scrape Dojo:

1. Set `UMASK=002` to allow group write access
2. Make sure the other service uses the same `PGID`
