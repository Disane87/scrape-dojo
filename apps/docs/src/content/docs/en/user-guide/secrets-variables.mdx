Passwords, API keys, login credentials -- stuff like that doesn't belong in plain text in your scrape configuration. Scrape Dojo has a well-thought-out system for encrypted secrets and flexible variables built right in. Let's take a look at how it works.

In short: secrets are for sensitive data, variables are for everything else. You can use both conveniently in your scrapes via template syntax.

## 🔒 Secrets

Secrets are stored encrypted in the database using **AES-256-CBC**. They're ideal for:

- Passwords and login credentials
- API keys and tokens
- OAuth client secrets
- Other confidential information

### 🔧 Creating Secrets

#### Via the UI

1. Open settings (gear icon)
2. Navigate to the "Secrets" tab
3. Click "New Secret"
4. Enter name, value, and an optional description
5. Save -- the value is automatically encrypted

#### Via the API

```bash
POST /api/secrets
Content-Type: application/json

{
  "name": "portalPassword",
  "value": "secret-password",
  "description": "Login password for the portal"
}
```

<Aside type="note">
The value is **never** returned in the API response. You only see metadata like name, description, and creation date.
</Aside>

#### Via Environment Variables

```bash
SCRAPE_DOJO_SECRET_PORTAL_PASSWORD=secret-password
SCRAPE_DOJO_SECRET_API_TOKEN=abc123xyz
```

On startup, Scrape Dojo automatically syncs these into the database. The naming is converted to camelCase:
- `SCRAPE_DOJO_SECRET_PORTAL_PASSWORD` becomes `portalPassword`

### 📋 Managing Secrets (CRUD)

```bash
# List all secrets (without values)
GET /api/secrets

# Get a single secret (without value)
GET /api/secrets/{id}

# Update a secret
PUT /api/secrets/{id}
{ "value": "new-value", "description": "Updated description" }

# Delete a secret
DELETE /api/secrets/{id}
```

### 🔗 Linking Secrets to Workflows

Secrets can be linked to specific workflows to restrict access. You can specify a workflow ID during creation.

### 🛠️ Using Secrets in Configurations

In your scrape configurations, you reference secrets simply via template syntax:

```jsonc
{
  "action": "type",
  "name": "Enter password",
  "params": {
    "selector": "input[type='password']",
    "text": "{{secrets.portalPassword}}"
  }
}
```

<Aside type="caution">
Secrets are only resolved at runtime. The configuration file only contains the placeholder -- the actual value never leaves the encrypted database until it's needed.
</Aside>

## 📦 Workflow Variables

Variables are defined in the `metadata.variables` array of a scrape configuration. When you start a scrape via the UI, Scrape Dojo automatically shows you an input form for the defined variables.

### 📋 Variable Types

| Type | Description | UI Element |
|------|-------------|------------|
| `string` | Free text input | Text field |
| `password` | Hidden input | Password field |
| `number` | Numeric value | Number field |
| `boolean` | True/false | Checkbox |
| `email` | Email address | Email field |
| `url` | URL input | URL field |
| `select` | Selection from options | Dropdown |

### 📝 Variable Definition

```jsonc
{
  "metadata": {
    "variables": [
      {
        "name": "targetYear",
        "label": "Target Year",
        "type": "number",
        "default": 2025,
        "required": true,
        "description": "Which year should be processed?"
      },
      {
        "name": "region",
        "label": "Region",
        "type": "select",
        "default": "de",
        "options": [
          { "value": "de", "label": "Germany" },
          { "value": "at", "label": "Austria" },
          { "value": "ch", "label": "Switzerland" }
        ]
      },
      {
        "name": "maxPages",
        "label": "Maximum Pages",
        "type": "number",
        "default": 10,
        "placeholder": "e.g. 10"
      },
      {
        "name": "email",
        "label": "Email",
        "type": "email",
        "secretRef": "portalEmail",
        "description": "Loaded from secret"
      }
    ]
  }
}
```

<Aside type="tip">
Variables with `secretRef` are automatically populated with the value of the referenced secret. The user can still override the value in the UI dialog.
</Aside>

### 🛠️ Using Variables in Templates

```jsonc
{
  "action": "navigate",
  "name": "Open page",
  "params": {
    "url": "https://example.com/data?year={{variables.targetYear}}&region={{variables.region}}"
  }
}
```

## ⚡ Runtime Variables

Runtime variables are passed when starting a scrape -- either via the UI dialog or the API body:

```bash
POST /api/scrape/my-workflow
Content-Type: application/json

{
  "variables": {
    "targetYear": 2023,
    "maxPages": 5
  }
}
```

## 💾 Database Variables

Global or workflow-specific variables stored via the API:

```bash
# Create a variable
POST /api/variables
{
  "name": "defaultYear",
  "value": "2025",
  "scope": "global",
  "description": "Default year for all scrapes"
}

# Get all variables
GET /api/variables

# Global variables
GET /api/variables/global

# Workflow-specific variables
GET /api/variables/workflow/{workflowId}

# Update a variable
PUT /api/variables/{id}
{ "value": "2026" }

# Delete a variable
DELETE /api/variables/{id}
```

Database variables can also be defined via environment variables with the prefix `SCRAPE_DOJO_VAR_`:

```bash
SCRAPE_DOJO_VAR_DEFAULT_YEAR=2025
SCRAPE_DOJO_VAR_API_ENDPOINT=https://api.example.com
```

## 🔢 Resolution Priority

When a variable is defined in multiple places, this order applies (highest first):

1. **Runtime** -- values passed at start via UI dialog or API body
2. **Database** -- values stored via `/api/variables` or `SCRAPE_DOJO_VAR_*`
3. **Config Defaults** -- default values defined in `metadata.variables[].default`

### 💡 Example

```bash
# Environment variable
SCRAPE_DOJO_VAR_TARGET_YEAR=2024
```

```jsonc
// Workflow configuration
{
  "metadata": {
    "variables": [
      { "name": "targetYear", "type": "number", "default": 2025 }
    ]
  }
}
```

```bash
# API call with runtime variable
POST /api/scrape/my-workflow
{ "variables": { "targetYear": 2023 } }
```

Result: `{{variables.targetYear}}` evaluates to **2023** (runtime wins).

Without a runtime variable, the value would be **2024** (database), and without a database variable, it would be **2025** (config default).

## 💾 storeData with persist

The `storeData` action can persist data between runs:

```jsonc
{
  "name": "save-results",
  "action": "storeData",
  "params": {
    "key": "myWorkflow.lastResults",
    "value": "{{previousData.transformedData}}",
    "persist": true
  }
}
```

With `persist: true`, the data is retained across multiple executions. Stored data is accessible via `{{storedData.myWorkflow.lastResults}}` in subsequent runs.

## 🔐 Encryption

### 🔐 Algorithm

Secrets are encrypted with **AES-256-CBC**:
- Key length: 256 bits (32 bytes)
- The encryption key is configured via `SCRAPE_DOJO_ENCRYPTION_KEY`
- Each secret receives its own initialization vector (IV)

### 🔑 Generating a Key

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

The generated 64-character hex string is set as `SCRAPE_DOJO_ENCRYPTION_KEY` in the `.env` file.

<Aside type="caution">
**Important notes:**
- **Never** commit the encryption key to a repository
- Store the key securely in a password manager
- If the key is lost, all stored secrets are **irrecoverably lost**
- For key rotation, all secrets must be recreated after changing the key
</Aside>
